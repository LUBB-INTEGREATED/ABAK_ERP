## RV-3 · P1 · security · (epic0)

**Location:** `packages/api/src/modules/files/files.controller.ts:97-112 (with files.module.ts FileInterceptor config)`

**Defect:** The upload endpoint buffers the entire request body into RAM before any size check, so any authenticated user can OOM the process by posting a multi-GB file.

**Fix:** Pass a hard limit to multer so it aborts mid-stream: FileInterceptor('file', { limits: { fileSize: MAX_UPLOAD_BYTES, files: 1 } }). Keep MaxFileSizeValidator as defense-in-depth. main.ts sets no global multipart body cap, so the interceptor limit is the only line of defense.

**Verified:** Confirmed real and reachable. At packages/api/src/modules/files/files.controller.ts:97 the upload endpoint uses @UseInterceptors(FileInterceptor('file')) with NO options. Multer 2.1.1 source (node_modules/.pnpm/multer@2.1.1/.../index.js lines 12-20) proves that with no `storage`/`dest` it defaults to memoryStorage() and sets this.limits = options.limits = undefined. That undefined limits is passed to busboy (make-middleware.js:129); the LIMIT_FILE_SIZE abort (line 211) only fires when limits.fileSize is set, so with no fileSize the file stream is read to completion and the memory-storage engine accumulates every chunk into a heap Buffer with no bound. The MaxFileSizeValidator inside ParseFilePipe (controller line 102) only inspects file.size AFTER the interceptor has fully buffered the body — too late to prevent the heap growth. No mitigation exists upstream: no MulterModule.register with limits anywhere in src (only this one FileInterceptor usage), main.ts sets no bodyParser/json/payload limit and no rawBody, and there is no nginx/Caddy reverse-proxy config in the repo. The route is guarded by a global JwtAuthGuard (app.module.ts:70) and the upload handler is NOT @Public() (only the GET :id/raw route is), so it is reachable by any authenticated user. Zero tests cover the files upload path (no matching .spec.ts/.e2e-spec.ts). The only minor inaccuracy in the report is wording: memoryStorage concatenates streamed chunks into a growing Buffer rather than allocating the full multi-GB Buffer in one shot, but the result is the same unbounded heap exhaustion / process crash. Proposed fix (pass limits: { fileSize: MAX_UPLOAD_BYTES, files: 1 } to FileInterceptor so multer aborts mid-stream, keeping MaxFileSizeValidator as defense-in-depth) is correct. P1: any logged-in user can OOM the single API process.

## RV-5 · P2 · concurrency · (seam)

**Location:** `packages/api/src/modules/rfqs/rfqs.service.ts:300-372 (startPricing tx, esp. L365-368)`

**Defect:** Two concurrent startPricing calls each read quoteId=null outside the tx, each create a quote+sections inside the tx, and the final tx.rfq.update has no quoteId-null precondition — so the second writer overwrites the link and orphans the first quote (the rfq.quoteId @unique constraint does NOT catch it because both updates touch the same single row with distinct values).

**Fix:** Make the link conditional and detect the loser: replace the final update with `const r = await tx.rfq.updateMany({ where: { id, quoteId: null }, data: { quoteId: quote.id, status: PRICING } }); if (r.count === 0) throw new ConflictException(); ` so the second tx rolls back (deleting its just-created quote+sections via the rollback), then re-read and return the winner. The commit message's claim that rfq.quoteId @unique structurally enforces double-click safety is false for the concurrent case.

**Verified:** Confirmed real and reachable at packages/api/src/modules/rfqs/rfqs.service.ts:300-372. The idempotency guard (L324-331) reads rfq.quoteId from a findUnique performed OUTSIDE the $transaction (L302), and the final tx.rfq.update (L365-368) uses where:{ id } with NO quoteId:null precondition. Schema confirms Rfq.quoteId is `String? @unique` (prisma/schema.prisma:1228). The reviewer's core argument is correct: the @unique constraint only prevents two DIFFERENT rfq rows from sharing a quoteId; in the double-create race both UPDATEs hit the SAME rfq row with DISTINCT quoteId values (qA vs qB), which is a write-write conflict, not a uniqueness violation. Provider is postgresql with no isolationLevel set → READ COMMITTED: the second UPDATE blocks on the first, then re-applies after commit (id predicate still matches) and overwrites quoteId qA→qB. Quote-A commits as an orphaned DRAFT quote with QuoteDepartmentSection rows and no RFQ back-reference. No FOR UPDATE / advisory lock / updateMany guard exists. Endpoint POST /rfqs/:id/start-pricing (controller L56-68) calls the service directly and is reachable; status check only blocks DECLINED/CANCELLED, so SUBMITTED/other non-terminal states pass. The 'subsequent dept manager joins' flow is a documented legitimate trigger. No test coverage: the only spec in the whole API package is quotes/quotes-revise-submit.spec.ts, which does not touch startPricing. Could not refute. Rated P2 not P1 because the trigger window is narrow (outside-tx read vs tx update, sub-second concurrent calls) and the impact is a dangling DRAFT quote + wrong link rather than silent financial loss or broad data destruction; the proposed updateMany({where:{id,quoteId:null}}) + count===0 → ConflictException fix is the correct remedy.

**Fix:** Pluralize/inflect the currency noun by the major count (1→singular Riyal/ريال, 2→dual, 3-10→ريالات, 11+→ريال) and the minor noun by the halala count with feminine agreement (1→هللة واحدة, 2→هللتان, 3-10→هللات, …). For EN, singularize 'Saudi Riyal'/'Halala' when the count is 1.

**Verified:** Confirmed by running the actual shipped code (both my logic replica and the compiled dist/lib/amount-in-words.js): amountInWords(1,{locale:'en'}) → 'One Saudi Riyals only' and amountInWords(1.01,{locale:'en'}) → 'One Saudi Riyals and one Halalas only' — both pluralized for a count of 1. AR: amountInWords(1.01,{locale:'ar'}) → 'واحد ريال سعودي وواحد هللة فقط لا غير', and هللة (feminine) gets no gender agreement. Root cause is genuine and reachable: the currency/minor nouns are hardcoded constants in CURRENCY_WORDS (lines 202-207: 'Saudi Riyals', 'Halalas', 'هللة') and the render path at lines 228 and 235-236 just concatenates the integer words with the fixed noun — there is no count-based inflection or gender agreement anywhere. This path is the one used for SAR quote PDFs (per the file docstring, frozen into the quote renderManifest at send), so wrong wording reaches client-facing documents. Untested: the spec 'handles single units' case (spec lines 50-56) only tests value 5, never 1, so the count-of-1 plural path has no test coverage. Two caveats lowering severity: (1) it is cosmetic — the numeric amount is correct, no data/financial-math/security impact; (2) the AR masculine 'واحد' form and the proposed number-after-noun reordering conflict with the test suite (lines 50-67, e.g. 'خمسة ريال سعودي', 'مليونان ريال سعودي'), which deliberately encodes the masculine financial form as the 'vetted contract', so part of the AR claim is an intended convention rather than a clear bug. The clear, uncontested defect is the EN plural-for-one (Riyals/Halalas when count==1) plus the هللة feminine-agreement gap. Real but P3 (cosmetic wording), not P2.
